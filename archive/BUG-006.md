# BUG-006: lazygit GIT_DIR env leak — gitsigns permanent context loss

**Status**: closed  
**Branch**: master (`nvim/init.lua`)  
**First appeared**: 2026-03  

---

## Symptoms

- After opening lazygit via `<leader>gg` while a `docs/` buffer is active, all `main/` buffers lose git signs and branch info in the statusline for the rest of the session.
- Corruption commits: lazygit committed `main/` file deletions onto `master` (commits `5cab8aa`/`26fd97b`, net-zero, later reversed).

## Root cause

`<leader>gg` sets `vim.env.GIT_DIR` + `vim.env.GIT_WORK_TREE` so lazygit.nvim can build `-w/-g` flags. These were never cleared after `vim.cmd("LazyGit")` returned. When lazygit closes, `sync.sync()` fires `gs.refresh()`, which reads `vim.env.GIT_DIR` (gitsigns `repo.lua:494`). With `GIT_DIR=worktrees/docs`, gitsigns resolves every `main/` buffer against the `docs/` toplevel, hits `"outside worktree"`, silently detaches — statusline shows nothing for the rest of the session.

The same leaked env caused the corruption commits: lazygit was operating on the `dev-docs` index while displaying `main/`'s files, making them all appear deleted.

## Fix applied

Clear both env vars immediately after `vim.cmd("LazyGit")`:

```lua
vim.cmd("LazyGit")
vim.env.GIT_DIR = nil       -- must be before sync.sync() / gs.refresh()
vim.env.GIT_WORK_TREE = nil
sync.sync()
```

lazygit.nvim reads the env vars synchronously during command construction before the terminal opens — they don't need to persist.

## Diagnostic checklist (if it reappears)

1. `:lua print(vim.env.GIT_DIR)` after closing lazygit — if non-nil, the clear was skipped or re-set somewhere. Fix: ensure the nil assignments are after `vim.cmd("LazyGit")`.
2. `:lua require("gitsigns").refresh()` manually — if signs return, gitsigns was using a stale context from step 1.
3. Check `lazygit.nvim` source (`~/.local/share/nvim-tdl/lazy/lazygit.nvim/lua/lazygit.lua:176`) — if the plugin was updated and now reads env vars *after* the terminal opens rather than before, the clear-immediately approach breaks. Pin the plugin version or switch to the final solution below.
4. Check `gitsigns` source (`repo.lua:494`) — `gitdir = gitdir or vim.env.GIT_DIR`. If this line changes behaviour, gitsigns may stop inheriting env vars entirely (good) or start caching them (bad).

## Why this keeps recurring

`GIT_DIR` + `GIT_WORK_TREE` are process-wide env vars. Setting them to satisfy lazygit is fundamentally at odds with every other git-aware tool in the same nvim process. Each fix patches one side and risks breaking the other. This is not a lazygit bug — it is a consequence of using env var injection as the integration mechanism.

## Final solution (not yet implemented)

Replace the entire `lazygit.nvim` integration with a raw terminal float. Build the lazygit command directly, never touch env vars:

```lua
local cmd = "lazygit -w " .. vim.fn.shellescape(work_tree)
          .. " -g " .. vim.fn.shellescape(git_dir)
-- open in a floating window via vim.api.nvim_open_win + vim.fn.termopen
```

Before that: replicate the lazygit.nvim float (~20 lines of Lua for the window, autocmd to close on `q`/`<Esc>`) and own the full integration. Nothing left to leak.

Last resort: dissolve the bare-repo + worktree structure. Merge `dev-docs` content into a `docs/` subdirectory on `master`. Lazygit works correctly with a single normal repo and no worktrees — all of BUG-001 through BUG-006 would not exist. The `docs/` orphan branch isolation is a nice property but has cost more engineering time than it saved.
