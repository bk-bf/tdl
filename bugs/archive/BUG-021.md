# BUG-021: LSP/treesitter semantic tokens bleed fg color over bufferline buffer name

**Status**: closed — fixed 2026-03-09  
**Branch**: main (`nvim/init.lua`)  
**First appeared**: 2026-03-09  
**Fixed**: 2026-03-09  

---

## Symptoms

- Buffer names in the bufferline tab bar are rendered in a random foreground color
  (e.g. green, orange) instead of the palette-defined color.
- The bleed only appears after LSP attaches to a buffer — a cold-start with no LSP
  shows correct colors; they go wrong as soon as the first LSP server comes up.
- Reopening nvim resets the colors, but they bleed again once LSP re-attaches.

## Root cause

Two mechanisms combine:

1. **Missing `nocombine = true`** on the bufferline highlight groups. Without it,
   Neovim's highlight combiner merges the bufferline group with whatever semantic-token
   highlight is active at that screen cell, producing an unexpected foreground color.

2. **Highlight groups reset after startup**. Colorscheme loads and certain LSP/treesitter
   initialisation steps call `vim.api.nvim_set_hl(0, ...)` or trigger `ColorScheme`,
   which wipes all custom highlight groups. The palette is applied once at startup but
   gets overwritten before or during LSP attach.

## Attempted fix — LspAttach trigger (reverted)

First attempt: add `LspAttach` to the autocmd that re-applies the palette:

```lua
vim.api.nvim_create_autocmd({ "ColorScheme", "LspAttach" }, {
  callback = function() apply_palette() end,
})
```

This caused LSP to stop working. `LspAttach` fires per-buffer during the LSP attach
sequence; calling `apply_palette()` mid-attach disrupts the initialisation handshake.
Reverted.

## Fix applied

Two-part fix in `nvim/init.lua`:

**Part 1** — add `nocombine = true` to every bufferline highlight group in
`apply_palette()` so the highlight combiner cannot merge in external colors:

```lua
vim.api.nvim_set_hl(0, "BufferLineFill",          { bg = c.bg,       nocombine = true })
vim.api.nvim_set_hl(0, "BufferLineBackground",     { fg = c.comment,  bg = c.tabline, nocombine = true })
-- ... (all bufferline groups)
```

**Part 2** — replace `LspAttach` with a one-time deferred `VimEnter` call that fires
500 ms after startup, after all plugins and LSP servers have fully initialised:

```lua
vim.api.nvim_create_autocmd("ColorScheme", {
  desc = "BUG-021: re-apply palette after colorscheme resets hl groups",
  callback = function() apply_palette() end,
})
vim.api.nvim_create_autocmd("VimEnter", {
  once = true,
  desc = "BUG-021: re-apply palette once after all plugins and LSP servers have initialised",
  callback = function() vim.defer_fn(apply_palette, 500) end,
})
```

`ColorScheme` covers any subsequent colorscheme reload during the session.
`VimEnter` + `defer_fn(…, 500)` covers the initial post-startup window where LSP
servers first attach and can overwrite highlight groups, without interfering with
the attach sequence itself.
