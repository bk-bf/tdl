# BUG-014: pressing Tab in treemux sidebar opens file inside sidebar pane instead of nvim pane

**Status**: closed — fixed 2026-03-09  
**Branch**: main (`nvim-treemux/treemux_init.lua`)  
**First appeared**: 2026-03-09  

---

## Symptoms

- User focuses the treemux sidebar pane.
- Pressing `<Tab>` opens a file (or cycles buffers) inside the treemux pane itself rather
  than in the main editor pane.
- The sidebar nvim instancre("sync").sync()
now shows a file buffer instead of nvim-tree.
- Spamming `ZQ` closes the rogue buffers and eventually returns to the nvim-tree view,
  or the sidebar can be toggled closed and re-opened with `prefix+Tab`.

## Root cause

`<Tab>` is globally bound to `BufferLineCycleNext` in `nvim/init.lua`:

```lua
{ "<Tab>", "<cmd>BufferLineCycleNext<cr>", desc = "Next tab" },
```

The treemux sidebar runs a **separate nvim process** (`NVIM_APPNAME=treemux`) that sources
`treemux_init.lua`. That init file bootstraps lazy.nvim and loads the full plugin set,
which includes bufferline. As a result, the `<Tab>` → `BufferLineCycleNext` keybind is
active inside the sidebar nvim instance. When the user presses `<Tab>` while focused on
the sidebar, bufferline cycles to the next buffer in the sidebar's own tab bar — which
opens a file buffer directly in the sidebar pane.

nvim-tree's default `<Tab>` binding (preview file) may also contribute depending on load
order, but the bufferline binding is the primary cause.

## Impact

Low severity — the sidebar can be restored via `ZQ` spam or `prefix+Tab` toggle. But it
is confusing UX: `<Tab>` is used in many contexts (completion, navigation) and silently
hijacking the sidebar pane breaks the mental model that the sidebar is read-only/navigation-only.

## Fix applied (Option A)

Added to `treemux_init.lua` after `require("lazy").setup(...)`:

```lua
-- BUG-014: suppress <Tab> in the sidebar nvim so BufferLineCycleNext (loaded
-- via the full plugin set) cannot open file buffers inside the sidebar pane.
vim.keymap.set("n", "<Tab>", "<Nop>", { noremap = true, silent = true })
```

This is safe: the treemux nvim instance is navigation-only; cycling buffers there is
never a desired action.
