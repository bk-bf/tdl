# BUG-020: markdown-preview opens Firefox instead of default browser; writes mozilla/ into repo

**Status**: closed — fixed 2026-03-09  
**Branch**: master (`nvim/init.lua`)  
**First appeared**: 2026-03-09  
**Fixed**: 2026-03-09  

---

## Symptoms

- `<leader>mp` opens the markdown preview in **Firefox** rather than the system default browser (Zen).
- A `mozilla/firefox/<profile>/` directory is created inside `main/` (i.e. inside the repo working tree).
- The directory reappears every time a markdown preview is opened, even after manual deletion.

## Root cause

BUG-007's fix (`aid.sh`: set `XDG_CONFIG_HOME=$AID_DIR` so nvim finds its config at
`$AID_DIR/nvim`) had an unintended transitive effect:

1. nvim is spawned with `XDG_CONFIG_HOME=/path/to/aid/main`
2. nvim spawns the `markdown-preview.nvim` node server as a child process — it **inherits** the env
3. When the node server opens a URL, it calls `xdg-open` as a child process — which also **inherits** the env
4. `xdg-mime query default x-scheme-handler/http` reads mimeapps.list from `$XDG_CONFIG_HOME` — but there is no mimeapps.list in the aid repo, so it falls back to the system default, which is **Firefox** (not Zen, which is configured as default in the real `~/.config/mimeapps.list`)
5. Firefox starts with `XDG_CONFIG_HOME` still pointing into the repo, so it writes its profile to `$XDG_CONFIG_HOME/mozilla/` — inside `main/`

The full inheritance chain:
```
tmux (aid.sh sets XDG_CONFIG_HOME=$AID_DIR)
  └── nvim (inherits XDG_CONFIG_HOME=$AID_DIR/main)
        └── node server (inherits XDG_CONFIG_HOME=$AID_DIR/main)
              └── xdg-open (inherits XDG_CONFIG_HOME=$AID_DIR/main)
                    └── Firefox (no mimeapps.list at that path → fallback default)
                          └── writes profile to $XDG_CONFIG_HOME/mozilla/
```

Why Zen was not the fallback: `xdg-open` dispatches to `x-scheme-handler/http` via mimeapps.list. With `XDG_CONFIG_HOME` pointing into the aid repo (no mimeapps.list present), `xdg-mime` falls back to `/usr/share/applications/` defaults. The system-level default is Firefox, not Zen — Zen is only registered as default in the user-level `~/.config/mimeapps.list`.

## Diagnostic

```bash
# Confirms the dirty env:
tr '\0' '\n' < /proc/$(pgrep -f 'markdown-preview.*index.js')/environ | grep XDG_CONFIG_HOME
# → XDG_CONFIG_HOME=/path/to/aid/main

# Confirms the browser dispatch difference:
xdg-mime query default x-scheme-handler/http
# → zen.desktop  (correct — reads real ~/.config/mimeapps.list)

XDG_CONFIG_HOME=/path/to/aid/main xdg-mime query default x-scheme-handler/http
# → firefox.desktop  (wrong — no mimeapps.list in repo, falls back to system default)
```

## Fix applied

`nvim/init.lua`: set `vim.g.mkdp_browserfunc = "MkdpOpenBrowser"` and define a vimscript
function that strips all aid XDG overrides before calling `xdg-open`:

```lua
init = function()
  vim.g.mkdp_browserfunc = "MkdpOpenBrowser"
  vim.cmd([[
    function! MkdpOpenBrowser(url) abort
      call jobstart([
        \ 'env',
        \ '-u', 'XDG_CONFIG_HOME',
        \ '-u', 'XDG_DATA_HOME',
        \ '-u', 'XDG_STATE_HOME',
        \ '-u', 'XDG_CACHE_HOME',
        \ 'xdg-open', a:url
        \ ])
    endfunction
  ]])
end,
```

When `mkdp_browserfunc` is non-empty, the node server calls back into nvim
(`plugin.nvim.call(browserfunc, [url])`) instead of spawning `xdg-open` directly.
nvim then runs `MkdpOpenBrowser`, which uses `jobstart` with `env -u` to strip the
four aid XDG vars before handing off to `xdg-open`. The browser is spawned with the
real user XDG dirs, so:
- `xdg-mime` resolves to `zen.desktop` correctly
- Zen starts with `XDG_CONFIG_HOME=~/.config` and writes its profile to `~/.config/mozilla/` as expected

Committed as: `fix: strip aid XDG overrides before opening browser in markdown-preview`

## Post-fix observation

After applying the fix, `mozilla/` continued to reappear briefly. Investigation showed
this was **not** a regression: a pre-fix Firefox instance (opened before the aid restart
that loaded the new config) was still running with the dirty `XDG_CONFIG_HOME` and
periodically writing to `datareporting/` in its profile. Once that stale process was
closed, `mozilla/` stopped reappearing entirely. New `<leader>mp` invocations correctly
open Zen and produce no `mozilla/` in the repo.
