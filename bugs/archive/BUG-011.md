# BUG-011: `.aidignore` changes not reflected in Telescope until nvim restart

**Status**: closed  
**Roadmap**: T-021  
**Branch**: master (`nvim/lua/aidignore.lua`, `nvim/init.lua`)  
**First appeared**: 2026-03-09  

---

## Symptoms

- Add or remove a pattern from `.aidignore`.
- The sidebar file tree and nvim-tree update immediately (live reload works).
- Open Telescope (`find_files` or `live_grep`) ÔÇö the changed pattern is not applied.
  Files that should be hidden still appear; files that should reappear are still absent.
- Restarting nvim fixes it. No in-session workaround exists.

## Root cause

`telescope.setup()` is called exactly once, inside Telescope's lazy `config` function
(`nvim/init.lua:471ÔÇô481`), which lazy.nvim runs a single time when the plugin first loads.
At that point `aidignore.patterns().telescope` is read and baked into
`opts.defaults.file_ignore_patterns`, which is then written into
`require("telescope.config").values` via `config.set_defaults()`.

`require("telescope.config").values` is a Lua module singleton ÔÇö a plain table held in
the nvim process's Lua state. Every picker reads `values.file_ignore_patterns` at
invocation time. **Updating that table at runtime would immediately affect all subsequent
Telescope calls** ÔÇö no re-source, no restart required.

The gap: `aidignore`'s live-update path (`_apply_to_nvimtree`, called from both the
`vim.uv` fs_event watcher and `reset()`) never touches Telescope's config. All three
trigger paths leave Telescope stale:

| Trigger | nvim-tree updated | Telescope updated |
|---------|:-----------------:|:-----------------:|
| `.aidignore` saved (fs_event watcher) | Ô£ô | Ô£ù |
| `DirChanged` autocmd ÔåÆ `reset()` | Ô£ô | Ô£ù |
| `<leader>R` ÔåÆ `sync.reload()` ÔåÆ `reset()` | Ô£ô | Ô£ù |

### Why `<leader>R` (full reload) also doesn't fix it

`sync.reload()` calls `source $MYVIMRC`, which re-executes `init.lua`. However lazy.nvim
tracks which plugin `config` functions have already run and does not re-execute them on
re-source ÔÇö so Telescope's `config` block (and the `file_ignore_patterns` assignment
inside it) is skipped. `reset()` is called afterwards but only calls `_apply_to_nvimtree`.

## Proposed fix

Add `_apply_to_telescope()` to `aidignore.lua` and call it alongside
`_apply_to_nvimtree()` in every update path.

```lua
local function _apply_to_telescope()
  local ok, cfg = pcall(require, "telescope.config")
  if not ok then return end
  local base = { "^%.git[/\\]" }
  for _, p in ipairs(M.patterns().telescope) do
    table.insert(base, p)
  end
  cfg.values.file_ignore_patterns = base
end
```

`require("telescope.config").values` is a module-level table (not a file on disk ÔÇö it
lives entirely in the nvim process's Lua state). Assigning to it is the same mechanism
Telescope itself uses internally: `config.set_defaults()` writes into this exact table.
The assignment is instantaneous and takes effect on the very next Telescope invocation.

This is consistent with the existing `_apply_to_nvimtree` approach, which also mutates
a private but stable internal table (`explorer.filters.ignore_list`) rather than calling
the public `setup()` API. The stability reasoning is the same: `config.values` has been
the authoritative config store since telescope.nvim's initial architecture and is
referenced throughout the codebase.

### Cleanup: remove duplicate pattern-building from `init.lua`

The Telescope `config` block in `init.lua:471ÔÇô481` manually reads
`aidignore.patterns().telescope` and passes it to `telescope.setup()`. Once
`_apply_to_telescope()` handles this at runtime, the manual merge is redundant. Replace
it with a call to `aidignore.reset()` (or `_apply_to_telescope()` directly) after
`telescope.setup(opts)`, so the startup path is identical to the live-update path:

```lua
config = function(_, opts)
  require("telescope").setup(opts)
  require("aidignore").reset()  -- applies nvim-tree + Telescope patterns uniformly
end,
```

This removes the duplicated `base`/`file_ignore_patterns` construction and ensures
startup and live-update are always in sync.
