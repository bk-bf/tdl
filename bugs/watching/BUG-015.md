# BUG-015: E5560 "writefile must not be called in a fast event context" after lazygit commit

**Status**: investigating — needs reproduction  
**Branch**: main (`nvim/lua/sync.lua`, possibly `vim-tpipeline`)  
**First appeared**: 2026-03-09  

---

## Symptoms

- User opens lazygit via `<leader>gg`.
- User stages files and presses `c` to commit (lazygit's built-in commit panel, no external editor).
- After the commit completes, `:messages` contains:

  ```
  E5560: writefile must not be called in a fast event context
  ```

- The error is intermittent — it did not reproduce in a follow-up session.
- No visible disruption: lazygit continues working, nvim does not crash.
- Full stack trace was never captured.

## Investigation status

Candidates for `writefile` in a fast-event context were audited across all installed plugins:

| Call site | Context | Verdict |
|-----------|---------|---------|
| `tpipeline#cleanup()` — `autoload/tpipeline.vim:293` | `VimLeavePre` autocmd | Safe (synchronous event) |
| `tpipeline#cautious_cleanup()` — `tpipeline.vim:313` | 45 ms `timer_start` callback | Safe (timer callbacks are not fast-event) |
| `undotree` — `autoload/undotree.vim:1172,1175` | Only fires on undo diff view | Not implicated |
| `lazy.nvim build.lua:38` | Build step | Not implicated |
| `persistence.nvim` | `VimLeavePre` only, uses `mksession` not `writefile` | Not implicated |

No direct call to `vim.fn.writefile` was found in our own code (`sync.lua`, `aidignore.lua`, `init.lua`).

## Leading hypothesis

`watch_buf()` (added in T-014/BUG-009 fix) registers `vim.uv.new_fs_event` watchers on each
open buffer's parent directory. When lazygit commits, git writes to `.git/` — which on Linux
(inotify) is **not** watched because our watchers are non-recursive and `.git/` is a subdirectory
of the project root, not a direct child of any open buffer's directory.

However, there is a narrower scenario: if the user has a file open directly in the **project
root** (e.g. `README.md`, `go.mod`), the watcher on the project root directory would fire when
`.git/COMMIT_EDITMSG` is written — `.git/` is a direct child of root, and inotify does fire
for modifications within direct children that are directories... actually this depends on inotify
semantics: `IN_CREATE`/`IN_MODIFY` on a directory entry fires for direct children only, not
for writes inside subdirectories. Requires empirical confirmation.

If the watcher does fire → `sync()` is called via `vim.schedule_wrap` (safe) → gitsigns
refreshes → mini.statusline recomputes → tpipeline's `OptionSet statusline` autocmd fires.
None of these are fast-event contexts. The chain appears safe.

Alternative hypothesis: the error is emitted by the **inner nvim** spawned as `$EDITOR` during
a `C` (capital-C) commit that the user may have used rather than `c`, or during a previous
session, and was still present in `:messages` from that earlier context.

## Next steps

1. **Capture the full stack trace**: on next occurrence, immediately run `:messages` and record
   all lines including any `stack traceback:` section. This will identify the exact call site.
2. **Check if `.git/` parent-dir watcher fires**: add a temporary `vim.notify` inside
   `watch_buf()`'s callback to confirm whether the watcher fires during a lazygit commit.
3. **Check inotify semantics for subdirectory writes**: verify whether a watcher on `/project/`
   receives an event when `/project/.git/COMMIT_EDITMSG` is written (direct child dir modified).
4. **Disable `watch_buf()` temporarily**: if the error is reproducible, toggle the watcher off
   to confirm whether it is the trigger.

## Context

- `vim.schedule_wrap` is used in all `vim.uv` callbacks in our code — calls through it are in a
  safe (non-fast) context and should not trigger E5560.
- `tpipeline_focuslost = 1` (default) means `FocusLost` triggers `deferred_cleanup()` — a
  45 ms timer callback that calls `chansend()`, not `writefile`. This is safe.
- `tpipeline#cleanup()` calls `writefile` only from `VimLeavePre`, which fires when a nvim
  instance exits — not during a commit.
- BUG was not reproduced in the investigation session immediately following the report.
  Tracking here to capture data on next occurrence.
