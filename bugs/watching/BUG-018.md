# BUG-018: bufferline tab bar does not show opened files even though they are loaded in nvim

**Status**: fixed — monitoring  
**Branch**: main (`nvim/init.lua`, bufferline config)  
**First appeared**: 2026-03-09  
**Fixed**: 2026-03-09

---

## Symptoms

- A file is opened from the treemux sidebar after having been closed earlier in the
  same session.
- The buffer is loaded and the file contents are visible in the editor pane.
- The tab bar (bufferline) shows no tab for that buffer.
- Running ZZ / ZQ / `:q` (full nvim exit → restart loop respawns it) clears the
  problem for the next session.

## Root cause

`:bdelete` — the default close action for the bufferline `x` button and `<leader>q`
— marks a buffer **unlisted and unloaded** but **keeps its entry** in nvim's buffer
table.  `:bwipe` would remove the entry entirely; `:bdelete` does not.

The dedup helper `_remote_bufnr()` in `treemux_init.lua` uses `vim.fn.bufnr(path)` to
check whether a file is already open before deciding to switch to it (`buffer N`) vs.
open it fresh (`tabnew`).  `bufnr()` returns **any** buffer matching that path,
including unlisted (bdelete'd) ghosts.

**Trigger sequence:**

1. User opens file from sidebar → `tabnew` → listed, loaded buffer → tab visible.
2. User closes the tab via the `x` button or `<leader>q` → `:bdelete` → buffer
   becomes **unlisted + unloaded**, ghost entry persists in nvim's buffer table.
3. User opens the same file from the sidebar again.
4. `_remote_bufnr()` calls `bufnr(path)` → finds the ghost → returns e.g. buffer 3.
5. Sends `buffer 3` to the editor nvim via RPC.
6. Nvim switches to buffer 3 and reloads the file from disk, but `buflisted` remains
   **false** — nvim never sets `buflisted` when switching to an existing buffer.
7. Bufferline filters on `buflisted=true` → ignores the buffer → no tab appears.

**Confirmed by live RPC inspection**: `getbufinfo()` showed `listed=0, loaded=0` on
affected files; manually setting `buflisted=1` caused the tab to appear immediately.  
**Confirmed by user reproduction**: clicking `x`, then reopening the same file from
the sidebar, reliably triggered the missing tab.  Full nvim exit (ZZ/ZQ) cleared all
ghost entries and resolved the problem for the next session.

## Fix

Use `:bwipe` instead of `:bdelete` everywhere a tab is closed. `bwipe` removes the
buffer entry entirely — `bufnr()` returns -1 on the next open, the dedup check falls
through, and `tabnew` opens a fresh listed buffer that bufferline renders normally.

**bufferline `close_command`** (`init.lua`, bufferline opts) — overrides the default
`bdelete! %d` for the `x` button and right-click:

```lua
close_command       = "bwipe! %d",
right_mouse_command = "bwipe! %d",
```

**`<leader>q`** (`init.lua`, bufferline keys):

```lua
{ "<leader>q", "<cmd>bwipe<cr>", desc = "Close tab" },
```

## Earlier fix (likely redundant, kept as belt-and-suspenders)

Before the root cause was identified, `BufEnter` was added to the bufferline redraw
autocmd on the theory that `redrawtabline` was not being called after certain RPC paths:

```lua
vim.api.nvim_create_autocmd({ "BufAdd", "BufEnter", "TabNew" }, {
  callback = function() vim.cmd("redrawtabline") end,
})
```

This was never the real fix — bufferline was rendering correctly; it just had nothing
to render because `buflisted=false`.  The autocmd is harmless and cheap so it remains,
but it does not address the root cause.
