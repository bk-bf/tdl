# BUG-018: bufferline tab bar does not show opened files even though they are loaded in nvim

**Status**: fixed — monitoring  
**Branch**: main (`nvim/init.lua`, bufferline config)  
**First appeared**: 2026-03-09  
**Fixed**: 2026-03-09 (root cause confirmed and resolved 2026-03-09)

---

## Symptoms

- A file is opened in nvim from the treemux sidebar.
- The buffer is loaded and the file contents are visible in the editor pane.
- The tab bar (bufferline) does not show a tab for that buffer.
- The missing tab may appear after switching buffers, pressing `<Tab>`, or after a
  second file is opened.
- Closing and reopening nvim entirely (ZZ / ZQ / `:q`) clears the problem.

## Root cause

### True root cause (confirmed 2026-03-09): bdelete ghost buffers

`bdelete` (and bufferline's default `close_command = "bdelete! %d"`) marks a buffer
**unlisted and unloaded** but keeps its entry in nvim's buffer table.  `:bwipe` would
remove it entirely; `:bdelete` does not.

The dedup helper `_remote_bufnr()` in `treemux_init.lua` calls `vim.fn.bufnr(path)` to
check whether a file is already open so it can switch to the existing tab instead of
opening a duplicate.  `bufnr()` returns **any** buffer with that path — including
unlisted (bdelete'd) ghosts.

Trigger sequence:

1. User opens file → `tabnew` → listed, loaded buffer, tab visible in bufferline.
2. User closes the tab via the bufferline `x` button or `<leader>q`
   → `:bdelete` → buffer becomes **unlisted + unloaded**, ghost entry persists.
3. User opens the same file again from the sidebar.
4. `_remote_bufnr()` calls `bufnr(path)` → finds the ghost → returns e.g. buffer 3.
5. `transport.exec("buffer 3")` is sent to the editor nvim.
6. Nvim switches to buffer 3 and reloads the file from disk, but `buflisted` stays
   **false** because switching to an existing buffer never sets it.
7. Bufferline only renders buffers with `buflisted=true` → tab bar ignores it.

**Confirmed by live RPC inspection**: `getbufinfo()` showed `listed=0, loaded=0` for
affected files; setting `buflisted=1` manually caused the tab to appear immediately.
**Confirmed by user testing**: closing and reopening the same file after clicking `x`
reproduced the missing tab reliably.  Running ZZ/ZQ/`:q` (which fully exit nvim,
clearing all ghost entries) eliminated the problem.

### Earlier partial diagnosis (BufEnter redraw)

Before the root cause was identified, an intermediate fix added `BufEnter` to the
bufferline redraw autocmd.  That fix is still correct and harmless — it ensures
bufferline redraws on every buffer switch — but it only addressed the redraw symptom,
not the ghost buffer root cause.

## Fix

### Fix 1 — bufferline `close_command` (`init.lua`, bufferline opts)

Override bufferline's default `bdelete! %d` with `bwipe! %d` for both the `x` button
and right-click:

```lua
close_command       = "bwipe! %d",
right_mouse_command = "bwipe! %d",
```

`bwipe` removes the buffer entry entirely.  `bufnr()` can never find it again, so the
dedup check correctly falls through to `tabnew` on the next open.

### Fix 2 — `<leader>q` keymap (`init.lua`, bufferline keys)

Changed `<cmd>bdelete<cr>` → `<cmd>bwipe<cr>` for the same reason.

### Earlier fix — BufEnter redraw autocmd (still present, still correct)

```lua
vim.api.nvim_create_autocmd({ "BufAdd", "BufEnter", "TabNew" }, {
  desc     = "BUG-018: force bufferline redraw after RPC-triggered buffer/tab open",
  callback = function() vim.cmd("redrawtabline") end,
})
```

Ensures bufferline re-renders on every buffer switch regardless of RPC path.  Cheap
and idempotent; no reason to remove it.
