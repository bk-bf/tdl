# BUG-024: Cross-session conversation switching — in-terminal session jump not achievable

**Status**: mitigated — see ADR-016 in `docs/DECISIONS.md`
**Branch**: `feature/orchestrator`  
**First appeared**: 2026-03-12  
**Mitigated**: 2026-03-12 (commit `e74d54b`)

---

## Original symptom

In `aid --mode orchestrator`, pressing Enter on a conversation owned by a different aid
session did nothing — the terminal did not jump to the foreign session.

## Root causes found and fixed

### Root cause 1 — `"not a tty"` stored as `AID_CALLER_CLIENT` (fixed in `b1ee4ee`)

The `tty` binary prints `"not a tty"` when stdin is not a terminal.  The init code ran
both resolution attempts in parallel, so the `tty` fallback raced with the tmux
`display-message` query.  Because `display-message` returned empty (no client attached
to the nav pane's session at startup), the bad string was stored.  `switch-client -c
"not a tty"` failed silently.

**Fix**: sequential resolution (tmux first, `tty` only as fallback); explicit
`"not a tty"` guard.

### Root cause 2 — `switch-client` cannot cross Hyprland workspaces (mitigated in `e74d54b`)

Even with a valid client tty, `switch-client` only switches the tmux session inside the
terminal process.  If that terminal window is on a different Hyprland workspace, the
window is not brought to focus — the switch happens invisibly in the background.

**Mitigation**: replaced `switch-client` with a Hyprland-aware strategy:
- If a terminal already has the session open: walk the tmux client PID up the process
  tree to find the Hyprland window, then `hyprctl dispatch focuswindow address:0x…`.
- If no terminal has the session: spawn `kitty -- tmux -L aid attach -t <session>` on
  the current workspace via `hyprctl dispatch exec`.

## What is still not solved

The ideal UX — pressing Enter in the nav pane and having the *same* terminal window
switch sessions, staying on the same workspace — requires `switch-client` to work
reliably.  This is blocked by the fundamental constraint that the nav pane's stdin is
not a tty (`respawn-pane`), so `AID_CALLER_CLIENT` cannot be resolved at startup from
`#{client_tty}`.

See **ADR-016** in `docs/DECISIONS.md` for the deferred research decision.
