# BUG-018: bufferline tab bar does not show opened files even though they are loaded in nvim

**Status**: open  
**Branch**: main (`nvim/init.lua`, bufferline config)  
**First appeared**: 2026-03-09  

---

## Symptoms

- A file is opened in nvim (e.g. from the sidebar, via `<leader>f`, or from the CLI).
- The buffer is loaded and the file contents are visible in the editor pane.
- The tab bar (bufferline) does not show a tab for that buffer — it either shows nothing,
  or only shows previously opened tabs that were already visible.
- The missing tab may appear after switching buffers, pressing `<Tab>`, or after a second
  file is opened.

## Root cause

**Identified (partially confirmed)**: The treemux sidebar opens files by sending a raw
Ex command (`:tabnew <path>`) to the main nvim instance via msgpack-RPC. The call path is:

1. User presses `<CR>` or `l` on a file in the nvim-tree sidebar (treemux pane).
2. `nvim-tree-remote.nvim` calls `transport.exec()` in
   `~/.local/share/nvim-treemux/lazy/nvim-tree-remote.nvim/lua/nvim_tree_remote/transport.lua`.
3. `transport.exec()` sends a `nvim_command("tabnew <file>")` call over the msgpack-RPC
   socket to the main nvim instance.
4. When the command arrives via RPC rather than a normal UI keypress, the `BufAdd` /
   `TabNew` events fire inside the RPC dispatch loop. bufferline's internal update handler
   is an autocmd on those events, but **the tabline is not redrawn** because `redrawtabline`
   is not called after the RPC command completes — nvim only redraws the tabline on its
   normal redraw cycle, which may not be entered promptly after an RPC-triggered command.
5. The result: the buffer and tab exist (`BufAdd` fired, `:buffers` shows the buffer) but
   bufferline's rendered tabline is stale.

**Ruled out**: render-timing of the startup 50 ms `defer_fn`. Closing and reopening the
same file does not fix the missing tab, so a one-shot startup delay is not the cause.

## Reproduction

Not yet reliably reproducible from an automated sequence. Observed intermittently when
opening files from the treemux sidebar or on a cold start with a file argument.

**Steps to attempt repro:**
1. Start `aid` in a directory with several files.
2. Open a file from the treemux sidebar (`<CR>` or `l`).
3. Observe the tab bar — check whether a tab appears for the opened file.
4. Open a second file. Check whether both tabs appear.
5. If a tab is missing: run `:buffers` to confirm the buffer is loaded; run
   `:set showtabline?` to confirm it is `2`; run `redrawtabline` manually to see
   if the tab appears (confirms the RPC-redraw theory).

## Fix direction

Add a `BufAdd` + `TabNew` autocmd in `init.lua` (inside the bufferline `config` function)
that calls `redrawtabline` on every new buffer or tab, regardless of how it was created:

```lua
vim.api.nvim_create_autocmd({ "BufAdd", "TabNew" }, {
  callback = function() vim.cmd("redrawtabline") end,
})
```

Also add `always_show_bufferline = true` to bufferline options so it never hides the bar
if `showtabline` is temporarily `1`.

## Next steps

- Implement the `BufAdd`/`TabNew` autocmd fix (see Fix direction above).
- Add `always_show_bufferline = true` to bufferline `options` block.
- Monitor over several sessions to confirm the intermittent miss no longer occurs.
- If still intermittent, add a `BufEnter` autocmd as a belt-and-suspenders fallback.
