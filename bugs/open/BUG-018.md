# BUG-018: bufferline tab bar does not show opened files even though they are loaded in nvim

**Status**: fixed — monitoring  
**Branch**: main (`nvim/init.lua`, bufferline config)  
**First appeared**: 2026-03-09  
**Fixed**: 2026-03-09

---

## Symptoms

- A file is opened in nvim (e.g. from the sidebar, via `<leader>f`, or from the CLI).
- The buffer is loaded and the file contents are visible in the editor pane.
- The tab bar (bufferline) does not show a tab for that buffer — it either shows nothing,
  or only shows previously opened tabs that were already visible.
- The missing tab may appear after switching buffers, pressing `<Tab>`, or after a second
  file is opened.

## Root cause

Three distinct RPC code paths reach the main nvim from the treemux sidebar, and the
original fix only covered one of them.

### Code paths

1. **`tabnew <file>` — new buffer** (`tabnew_follow_symlinks`, file not yet loaded)
   - Fires `TabNew` + `BufAdd`. Original autocmd covered this path.

2. **`buffer N` — dedup path** (`tabnew_follow_symlinks`, file already loaded in buffer N)
   - `treemux_init.lua` calls `transport.exec("buffer " .. bufnr)` to switch to the
     existing buffer instead of opening a duplicate tab.
   - Fires only `BufEnter`. **Original `{ "BufAdd", "TabNew" }` autocmd missed this.**

3. **`edit <file>` — neo-tree path** (`file_open_requested` handler)
   - For a brand-new buffer: fires `BufAdd` + `BufEnter`. Original autocmd covered this.
   - For a previously loaded (unlisted/hidden) buffer: fires only `BufEnter`.
     **Original autocmd missed this.**

The result in gap cases: the buffer is visible and `:buffers` shows it, but bufferline's
rendered tabline is stale because `redrawtabline` was never called.

**Ruled out**: render-timing of the startup 50 ms `defer_fn`. Closing and reopening the
same file does not fix the missing tab, so a one-shot startup delay is not the cause.

**Ruled out**: stale installed config. `aid.sh` sets `XDG_CONFIG_HOME=$AID_DIR` inline
on the nvim respawn command, so there is no separate installed copy that could be stale.

## Fix

### Primary fix (`init.lua:473`)

Added `BufEnter` to the bufferline autocmd event list so all three RPC code paths are
covered:

```lua
vim.api.nvim_create_autocmd({ "BufAdd", "BufEnter", "TabNew" }, {
  desc     = "BUG-018: force bufferline redraw after RPC-triggered buffer/tab open",
  callback = function() vim.cmd("redrawtabline") end,
})
```

`BufEnter` fires on every buffer switch regardless of origin. `redrawtabline` is cheap
and idempotent so the extra firings on normal keypresses are harmless. See ADR-015 for
the full rationale behind choosing this approach over unifying the send side.

### Secondary fix (`init.lua:457`)

`<leader>tb` (tab bar toggle) was setting `showtabline` without calling `redrawtabline`.
When toggling back on (`1 → 2`), the bar would reappear with potentially stale content
if buffers were opened while it was hidden. Added `redrawtabline` after the toggle.

## Reproduction

Not yet reliably reproducible from an automated sequence. Observed intermittently when
opening files from the treemux sidebar or on a cold start with a file argument.

**Steps to attempt repro:**
1. Start `aid` in a directory with several files.
2. Open a file from the treemux sidebar (`<CR>` or `l`).
3. Observe the tab bar — check whether a tab appears for the opened file.
4. Open a second file. Check whether both tabs appear.
5. If a tab is missing: run `:buffers` to confirm the buffer is loaded; run
   `:set showtabline?` to confirm it is `2`; run `redrawtabline` manually to see
   if the tab appears (confirms the RPC-redraw theory).
