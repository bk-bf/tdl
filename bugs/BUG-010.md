# BUG-010: opening an already-open file creates a duplicate tab

**Status**: closed — fixed in T-015  
**Branch**: main (`nvim-treemux/treemux_init.lua`)  
**First appeared**: 2026-03  
**Fixed**: 2026-03  

---

## Symptoms

- Open a file from the sidebar tree (`l` / `<CR>`).
- Open the same file again from the sidebar tree.
- The tab bar now shows two tabs for the same file instead of focusing the existing one.
- The buffer is loaded twice; both tabs display identical content.

## Root cause

The open path in the sidebar nvim is:

```
treemux_init.lua:tabnew_follow_symlinks()
  → nvim_tree_remote.remote_nvim_open(socket_path, "tabnew", path, tmux_opts)
    → transport.open(path, "tabnew", socket_path, 0)
      → transport.exec("tabnew <path>", socket_path, 0)
        → vim.rpcrequest(chan, "nvim_command", "tabnew <path>")
```

`transport.open()` (`nvim-tree-remote.nvim/lua/nvim_tree_remote/transport.lua:76–81`) constructs
a raw Ex command string (`"tabnew <path>"`) and fires it into the main nvim over msgpack-RPC.
Vim's `:tabnew` unconditionally opens a new buffer — it has no awareness of whether the target
path is already loaded. No deduplication check exists anywhere in the call chain.

The `duplicate_*` highlight groups defined in the bufferline config (`nvim/init.lua:429–431`) are
intended for files that share a filename across different directories — not for the same file
loaded twice. Their presence does not indicate intentional duplicate-tab support.

## Fix applied

`_remote_bufnr(socket_path, abs_path)` extracted to module scope in
`main/nvim-treemux/treemux_init.lua`. It opens a raw msgpack-RPC channel to `AID_NVIM_SOCKET`,
calls `bufnr(abs_path)` in the main nvim, and returns the buffer number or `-1`.

Both open paths now call it:

1. **`tabnew_follow_symlinks()`** (nvim-tree, `l`/`<CR>`/`<C-t>`/double-click): if `bufnr != -1`,
   issue `buffer <n>` instead of `tabnew`.
2. **neo-tree `file_open_requested` handler** (`t` key): same check added inside the
   `open_cmd == "tabnew"` branch — returns early with `{ handled = true }` after the switch.

`_remote_bufnr` is defined at module scope (above `nvim_tree_on_attach`) so both handlers share
the same implementation.
