# BUG-009: opencode file edits not reflected in nvim until user switches pane focus

**Status**: closed — T-014  
**Branch**: main (`nvim/lua/sync.lua`, `nvim/init.lua`, `tmux.conf`)  
**First appeared**: 2026-03  
**Fixed**: 2026-03  

---

## Symptoms

- opencode edits a file in the project; the buffer in the nvim pane shows stale content.
- The buffer updates correctly the moment the user clicks into or navigates to the nvim pane.
- No manual `:e` or `:checktime` is required — focus switch alone triggers the reload.
- The issue is 100% reproducible: every opencode edit is invisible in nvim until focus moves.

## Root cause

`sync.lua` triggers `checktime` (buffer reload) from a `FocusGained` autocmd. `FocusGained`
fires when the terminal emulator (or tmux) sends a focus-in event to the active process.

The critical constraint: **tmux only forwards focus events to the currently active pane**. When
opencode writes a file from its pane, the nvim pane is in the background — tmux never sends it
a focus event. `FocusGained` does not fire. `checktime` is never called. The buffer stays stale.

The user switching to the nvim pane makes it active, tmux sends the focus-in event, nvim fires
`FocusGained`, `sync.sync()` runs `checktime`, and the buffer reloads. This is the exact sequence
the user observes.

The delay is not a tmux race or a timing issue — it is architectural: the old design only pulled
file state on focus acquisition, which never happens while another pane holds focus.

### Why `autoread` alone does not fix it

`vim.opt.autoread = true` re-reads a file, but only when nvim itself detects a reason to check —
e.g. a shell command completes in the same nvim instance, or `checktime` is explicitly called.
Without an external trigger, `autoread` is dormant. It does not poll the filesystem.

### Why `pane-focus-in` alone does not fix it

A tmux `pane-focus-in` hook can push `--remote-send` into the nvim socket on every pane switch.
This helps when the user switches *away* from the opencode pane (e.g. to treemux), but it does
not help when the user stays in the opencode pane — the hook never fires for nvim while the user
sits in another pane. The buffer only reloads on the next pane switch, not immediately.

## Fix

`sync.watch_buf()` added to `sync.lua`: registers a `vim.uv.new_fs_event` watcher on the parent
directory of each open buffer. On any filesystem change in that directory, `sync.sync()` fires
immediately — no pane switch, no user action required.

Note: `vim.uv.new_fs_event` on Linux (inotify via libuv) does **not** support recursive
directory watching (`recursive = true` is silently a no-op). The watcher is therefore registered
per-buffer-directory, not on cwd. This is sufficient: opencode edits files that are already open
in nvim, so their directories are already watched.

`BufEnter` autocmd in `init.lua` calls `sync.watch_buf()` as new buffers are opened, so the
watch set grows automatically. `VimLeave` calls `sync.stop_watchers()` to clean up handles.

The `pane-focus-in` hook in `tmux.conf` is retained as belt-and-suspenders: it covers the case
where opencode creates a new file that isn't open in nvim yet (directory not yet watched).

## Files changed

| File | Change |
|------|--------|
| `nvim/lua/sync.lua` | Added `watch_buf()`, `stop_watchers()`, `_watchers` table |
| `nvim/init.lua` | `BufEnter` → `sync.watch_buf(ev.buf)`; `VimLeave` → `sync.stop_watchers()`; `VimEnter` → `sync.watch_buf(data.buf)` |
| `tmux.conf` | `pane-focus-in` hook updated to call `sync.sync()` (retained as belt-and-suspenders) |
