# BUG-009: opencode file edits not reflected in nvim until user switches pane focus

**Status**: closed — T-014  
**Branch**: master (`tmux.conf`, `nvim/lua/sync.lua`)  
**First appeared**: 2026-03  
**Fixed**: 2026-03  

---

## Symptoms

- opencode edits a file in the project; the buffer in the nvim pane shows stale content.
- The buffer updates correctly the moment the user clicks into or navigates to the nvim pane.
- No manual `:e` or `:checktime` is required — focus switch alone triggers the reload.
- The issue is 100% reproducible: every opencode edit is invisible in nvim until focus moves.

## Root cause

`sync.lua` triggers `checktime` (buffer reload) and `gitsigns.refresh()` from a `FocusGained`
autocmd in `nvim/init.lua`. `FocusGained` fires when the terminal emulator (or tmux) sends a
focus-in event to the active process.

The critical constraint: **tmux only forwards focus events to the currently active pane**. When
opencode writes a file from its pane, the nvim pane is in the background — tmux never sends it
a focus event. `FocusGained` does not fire. `checktime` is never called. The buffer stays stale.

The user switching to the nvim pane makes it active, tmux sends the focus-in event, nvim fires
`FocusGained`, `sync.sync()` runs `checktime`, and the buffer reloads. This is the exact sequence
the user observes.

The delay is not a tmux race or a timing issue — it is architectural: the current design only
pulls file state on focus acquisition, which never happens while another pane holds focus.

### Why `autoread` alone does not fix it

`vim.opt.autoread = true` re-reads a file, but only when nvim itself detects a reason to check —
e.g. a shell command completes in the same nvim instance, or `checktime` is explicitly called.
Without an external trigger, `autoread` is dormant. It does not poll the filesystem.

## Proposed fix

The fix must push a `checktime` into nvim from *outside* — without waiting for focus — whenever
opencode writes a file. Two viable approaches:

### Option A: tmux `after-` hook (preferred — no new deps)

tmux fires `after-<command>` hooks after every client command. A `after-send-keys` hook (or more
precisely a `pane-focus-in` hook on the nvim pane) can send a signal or run a command:

```bash
# In tmux.conf — after any pane in the window gets focus, tell nvim to checktime.
# Only fires for the nvim pane (matched by pane_id stored in $TDL_NVIM_SOCKET env).
set-hook -g pane-focus-in "run-shell 'nvim --server #{TDL_NVIM_SOCKET} --remote-send \"<cmd>silent! checktime<cr>\" 2>/dev/null || true'"
```

This fires every time any pane in the session gains focus — including when tmux itself switches
active pane programmatically (e.g. after opencode finishes). The `--remote-send` goes to the nvim
socket set in `$TDL_NVIM_SOCKET`.

Downside: `pane-focus-in` fires on every pane switch, not only on opencode writes. The `checktime`
cost is low (no-op if files are unchanged), but it adds one RPC round-trip per pane switch.

### Option B: `vim.uv` file-change watcher inside nvim

Register a `vim.uv.new_fs_event` watcher on the workspace root inside nvim. On any filesystem
event (write/rename) that matches a currently-open buffer, call `checktime` for that buffer only.

```lua
-- Sketch — in sync.lua or a dedicated watcher.lua
local watcher = vim.uv.new_fs_event()
watcher:start(vim.fn.getcwd(), { recursive = true }, vim.schedule_wrap(function()
  vim.cmd("silent! checktime")
end))
```

Downside: recursive `fs_event` on large repos is expensive on Linux (inotify watch count). Would
need path filtering (respect `.aidignore`, cap depth).

### Option C: tighten `FocusGained` to also cover `BufEnter` + `CursorHold`

A cheap interim patch: add `BufEnter` and `CursorHold` autocmds that call `checktime`. This does
not fix the "stale while opencode is writing" window, but it shortens it — any cursor movement
after opencode finishes will reload without requiring a pane switch.

```lua
vim.api.nvim_create_autocmd({ "FocusGained", "BufEnter", "CursorHold" }, {
  callback = function() vim.cmd("silent! checktime") end,
})
```

This is the lowest-effort option and meaningfully improves UX even if it doesn't fully solve it.

## Relationship to `sync.lua`

`sync.lua` is the right home for the fix — it already owns the `checktime` call and the
`FocusGained` hook registration belongs in `init.lua`. The change is:

1. Keep `FocusGained` as-is (correct and cheap).
2. Add one of the above mechanisms to trigger `checktime` *without* waiting for focus.
3. Option A requires a one-line addition to `tmux.conf` and a guard in `sync.lua` to handle
   the RPC path (or rely on `--remote-send` from the shell side entirely).

The TODO comment in `sync.lua:31-38` (RPC transport approach) is directly related — the same
msgpack-rpc channel that would push sidebar refreshes could also push `checktime` calls from
an external trigger.

## Affected users

All users — this affects every opencode interaction where opencode modifies files.
The workaround (switch focus to nvim after each opencode edit) is functional but breaks flow.
